Resumo executivo

O Radar Ofertas é um bot de coleta e publicação de ofertas do Mercado Livre com painel administrativo, filas BullMQ, PostgreSQL/Prisma e envio multi-canal (WhatsApp/Telegram). A base em src/ segue organização por domínio com separação clara entre regras de negócio, integrações e orquestração.

Após as Fases 1–4 de escalabilidade (pool de filas, multi-conta runtime, contas em Prisma, manager stateless), o projeto evoluiu de um MVP operacional para uma base sólida single-tenant com caminho claro para N contas. Os maiores gaps restantes são operacionais (spawn por conta, token Telegram) e de segurança (painel sem CSRF, auth opcional, XSS em atributos JS) — adequado para rede confiável/VPN, não para exposição pública sem hardening.

~102 arquivos src/ ~62 arquivos manager/ 20 testes unitários TS strict CI: tsc + test

1. Arquitetura — 8,0 / 10

Pontos fortes

  • Processos separados: collector (app.ts), workers por canal (worker.ts, worker-telegram.ts), manager — cada um com entry point e fila própria.
  • Contrato de canal: ChannelPublisher em channels/types.ts — sender genérico (jobs/sender.ts) sem if (whatsapp).
  • Delivery-first: dispatchOffer abre OfferDelivery antes de enfileirar — rastreabilidade e idempotência.
  • Fan-out multi-conta: enfileiramento por (canal, accountId) com filas nomeadas (getSenderQueueName).
  • Config runtime: score, brand, horários, templates em tabela settings com cache e fallback ENV.
  • Scraping híbrido: HTTP primeiro, Playwright fallback, circuit breaker — decisão documentada e implementada.
  • Manager como UI: MVC server-rendered reutiliza src/; regras de oferta não vivem no painel.

Pontos fracos

  • Login ML (Playwright) permanece stateful no manager — não replicável em cluster.
  • Telegram multi-conta incompleto: token ainda lê env.TELEGRAM_BOT_TOKEN global.
  • telegram-publisher.ts legado coexistindo com publisher-factory.ts.

4. Escalabilidade — 7,0 / 10

Já ok

  • já ok Pool de filas BullMQ: getQueue() reutilizável — sem overhead de criar/fechar Queue a cada enqueue.
  • já ok Filas por conta: offer-sender-{accountId} — isolamento de jobs por conta.
  • já ok Contas em Prisma: tabela accounts com validação Zod — base para FKs futuras.
  • já ok Manager stateless: MANAGER_CAN_SPAWN_WORKERS=false no Docker; status via Redis + owner.lock.
  • já ok QR/status no Redis: radar:connect:wa:{accountId} — painel replicável atrás de load balancer.
  • já ok Heartbeat worker: radar:worker:{channel}:{accountId} — detecção cross-process.
  • já ok Delivery por conta: unique (offer_id, channel, account_id) — fonte da verdade granular.
  • já ok Collector otimizado: Playwright pooled (1 Chromium/processo), auto-messages em scheduler separado, concurrency: 1 na fila.
  • já ok Collector sharded: job collect-orchestrate fan-out por fonte (collect-source) — paralelismo na fila; Playwright pooled.
  • já ok Pacing atômico: sender-pacing.ts via Redis (ou mutex in-process) — seguro com senderConcurrency > 1.
  • já ok Spawn por conta: painel sobe workers com WORKER_ACCOUNT_ID — um card por conta habilitada em Settings → Operações (dev local).
  • já ok Caches coerentes: invalidação via Redis pub/sub (cache-coherence) — accounts, queue-config e settings sincronizam entre réplicas; métricas centralizadas no Redis.

Gargalos e limites restantes

  • WhatsApp: 1 sessão Baileys = 1 processo — não escala horizontalmente no mesmo número.
  • Auth manager: MANAGER_TOKEN opcional — acesso livre se ausente.

5. Segurança — 5,5 / 10

O projeto foi desenhado como ferramenta operacional para uso local ou em rede confiável (VPN). Para exposição na internet pública, há débitos significativos que precisam ser endereçados antes de produção.

Pontos fortes

  • ORM parametrizado: Prisma em todo acesso a dados — sem SQL injection em regras de negócio. Raw queries só em preflight.ts com tagged templates.
  • Validação de entrada: Zod em config/env.ts e account-config.ts — shape tipado na leitura/gravação.
  • Escape HTML server-side: escapeHtml() em views do manager — mitiga XSS em conteúdo renderizado.
  • Path traversal bloqueado: manager/http/static.ts normaliza path e valida prefixo do diretório público.
  • Lock de sessão WhatsApp: owner.lock impede dois processos na mesma sessão Baileys — reduz hijack acidental.
  • Segredos fora do código: credenciais em .env (não commitado); .env.example sem valores reais.
  • Auth opcional documentada: MANAGER_TOKEN com Bearer ou query param — consciente do trade-off dev vs prod.

Recomendações prioritárias (segurança)

  1. MANAGER_TOKEN obrigatório em produção — validar no preflight profile manager; rejeitar boot sem token se NODE_ENV=production.
  2. CSRF tokens em todos os POSTs destrutivos — especialmente delete de ofertas e contas.
  3. Corrigir XSS em accounts.ts — escapar ' em escapeHtml ou remover onsubmit inline (usar data-* + JS).
  4. Migrar auth para cookie HttpOnly — eliminar ?token= na URL; propagar sessão nos forms.
  5. Limitar readFormBody — cap de 1 MB; retornar 413 se exceder.
  6. Redis/Postgres — não expor portas em produção; Redis com requirepass; rede interna Docker.
  7. Headers de segurança — CSP, X-Frame-Options, HSTS via reverse proxy (nginx/Caddy) ou middleware.

Conclusão

O Radar Ofertas apresenta arquitetura madura para o escopo atual — coleta ML, filas, envio multi-canal e painel administrativo — com decisões conscientes no código (delivery antes do enqueue, filas por canal/conta, affiliate link lazy, scraping híbrido). A organização é um dos pontos mais fortes do projeto.

A qualidade de código é boa no núcleo (src/), com TypeScript rigoroso e testes nos domínios críticos, mas perde pontos por dívidas de tipagem no manager/Baileys e cobertura ainda parcial em workers e rotas HTTP (helpers de jobs/, whatsapp/ e manager/ já têm testes unitários).

A escalabilidade melhorou significativamente com as Fases 1–4 e os itens marcados como já ok na seção 4 (filas por conta, manager stateless, collector sharded, pacing atômico, spawn por conta, caches coerentes). Limites reais restantes: WhatsApp é intrinsecamente single-process por sessão, e em produção/Docker workers adicionais ainda usam override ou terminal.

Em segurança, o projeto está adequado para uso em ambiente controlado (localhost, VPN, rede interna), mas não está pronto para exposição pública sem implementar CSRF, auth obrigatória, correção de XSS e isolamento de Redis/Postgres. A nota 5,5 reflete essa distinção: boas práticas no núcleo (Prisma, Zod, escape HTML), mas superfície de ataque significativa no painel administrativo.
