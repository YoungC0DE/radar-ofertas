Resumo executivo

O Radar Ofertas é um bot de coleta e publicação de ofertas do Mercado Livre com painel administrativo, filas BullMQ, PostgreSQL/Prisma e envio multi-canal (WhatsApp/Telegram). A base em src/ segue organização por domínio com separação clara entre regras de negócio, integrações e orquestração.

Após as Fases 1–4 e o hardening recente (collector sharded, spawn por conta, caches coerentes, tsc limpo, ESLint/Prettier, 111 testes), o projeto evoluiu de MVP operacional para base sólida multi-conta com gate de CI completo. Os maiores gaps restantes são segurança do painel (CSRF, auth opcional, XSS) e Telegram multi-conta — adequado para rede confiável/VPN, não para exposição pública.

~102 arquivos src/ ~62 arquivos manager/ 21 arquivos de teste · 111 casos TS strict CI: tsc + lint + format + test

1. Arquitetura — 8,0 / 10

Pontos fortes

  • Processos separados: collector (app.ts), workers por canal (worker.ts, worker-telegram.ts), manager — cada um com entry point e fila própria.
  • Contrato de canal: ChannelPublisher em channels/types.ts — sender genérico (jobs/sender.ts) sem if (whatsapp).
  • Delivery-first: dispatchOffer abre OfferDelivery antes de enfileirar — rastreabilidade e idempotência.
  • Fan-out multi-conta: enfileiramento por (canal, accountId) com filas nomeadas (getSenderQueueName).
  • Config runtime: score, brand, horários, templates em tabela settings com cache e fallback ENV.
  • Scraping híbrido: HTTP primeiro, Playwright fallback, circuit breaker — decisão documentada e implementada.
  • Manager como UI: MVC server-rendered reutiliza src/; regras de oferta não vivem no painel.

Pontos fracos

  • Login ML (Playwright) permanece stateful no manager — não replicável em cluster.
  • Telegram multi-conta incompleto: token ainda lê env.TELEGRAM_BOT_TOKEN global.
  • telegram-publisher.ts legado coexistindo com publisher-factory.ts.

5. Segurança — 5,5 / 10

O projeto foi desenhado como ferramenta operacional para uso local ou em rede confiável (VPN). Para exposição na internet pública, há débitos significativos que precisam ser endereçados antes de produção.

Pontos fortes

  • ORM parametrizado: Prisma em todo acesso a dados — sem SQL injection em regras de negócio. Raw queries só em preflight.ts com tagged templates.
  • Validação de entrada: Zod em config/env.ts e account-config.ts — shape tipado na leitura/gravação.
  • Escape HTML server-side: escapeHtml() em views do manager — mitiga XSS em conteúdo renderizado.
  • Path traversal bloqueado: manager/http/static.ts normaliza path e valida prefixo do diretório público.
  • Lock de sessão WhatsApp: owner.lock impede dois processos na mesma sessão Baileys — reduz hijack acidental.
  • Segredos fora do código: credenciais em .env (não commitado); .env.example sem valores reais.
  • Auth opcional documentada: MANAGER_TOKEN com Bearer ou query param — consciente do trade-off dev vs prod.

Recomendações prioritárias (segurança)

  1. MANAGER_TOKEN obrigatório em produção — validar no preflight profile manager; rejeitar boot sem token se NODE_ENV=production.
  2. CSRF tokens em todos os POSTs destrutivos — especialmente delete de ofertas e contas.
  3. Corrigir XSS em accounts.ts — escapar ' em escapeHtml ou remover onsubmit inline (usar data-* + JS).
  4. Migrar auth para cookie HttpOnly — eliminar ?token= na URL; propagar sessão nos forms.
  5. Limitar readFormBody — cap de 1 MB; retornar 413 se exceder.
  6. Redis/Postgres — não expor portas em produção; Redis com requirepass; rede interna Docker.
  7. Headers de segurança — CSP, X-Frame-Options, HSTS via reverse proxy (nginx/Caddy) ou middleware.

7. Confiabilidade e resiliência — 8,0 / 10

Pergunta central: se um componente cair, o que quebra de verdade e o que se recupera sozinho?

Pontos fortes

  • Delivery-first: OfferDelivery criado antes do enqueue — rastreio de pendente/enviado/falha por canal/conta.
  • Job IDs determinísticos: senderJobId, collectorSourceJobId — evita duplicata na fila.
  • Retries BullMQ: 5 tentativas, backoff 30s; DelayedError para pacing e janela operacional.
  • Circuit breaker ML: HTTP bloqueado após falhas; half-open com cooldown exponencial.
  • Pacing atômico: sender-pacing.ts via Redis Lua — seguro com concorrência > 1.
  • WhatsApp owner.lock: impede dois processos na mesma sessão; detecção de conflito (connectionReplaced).
  • Fan-out isolado: falha em um canal/conta no dispatchOffer não impede os demais.
  • Affiliate link com timeout: 10s no envio — sessão ML lenta não segura a fila indefinidamente.

Pontos fracos

  • Sem dead-letter queue explícita documentada — jobs falhos dependem do retry BullMQ + painel de filas.
  • accounts/repository.ts silencia erro de DB e cai para ENV — pode mascarar estado inconsistente.
  • Redis single point of failure — sem fallback se BullMQ indisponível (coleta/envio param).
  • Reconexão WhatsApp automática após disconnect pode entrar em loop se credenciais inválidas.

8. Performance e custo — 7,5 / 10

Pergunta central: quantas contas/fontes o hardware atual aguenta antes de degradar?

Pontos fortes

  • HTTP primeiro: scraping ML tenta HTTP antes de Playwright — menor CPU/RAM na maioria dos casos.
  • Browser pool: 1 Chromium compartilhado por processo collector (browser-pool.ts).
  • Collector sharded: jobs collect-source em paralelo (concurrency 2) por fonte.
  • Pacing configurável: delay entre envios via settings — protege rate limit WhatsApp/Telegram.
  • Pool de filas: getQueue() reutilizável — sem overhead de conexão a cada enqueue.
  • Affiliate lazy: link gerado sob demanda no envio com timeout — não bloqueia coleta.

Pontos fracos

  • Playwright ainda é o gargalo de RAM quando fallback dispara em muitas fontes.
  • 1 processo Baileys por conta WhatsApp — custo linear em memória com multi-conta.
  • Shard por fonte usa quota igual (floor(target/N)) — não proporcional ao tamanho do pool ML.
  • Sem profiling/benchmarks automatizados — limites de capacidade são empíricos.
  • Paginação no manager pode carregar listas grandes sem virtualização.

Pergunta central: o banco é fonte da verdade confiável em multi-conta e multi-canal?
